The Purely Functional Imperative Language in the Command Line Interface

Like all compiled languages, the Purely Functional Imperative Language needs a compiler. And, as many compilers use the Command Line Interface, a syntax must be established. This chapter is more of a set of guidelines on how to design this syntax than strict rules.

The basic syntax for this is [compiler name] [filepath]. For example, to compile foo.pfil with the Arbitrary Imaginary Compiler (aic), you would use aic foo.pfil. This would compile to a file named, for example, a.exe.

If one wishes to control the output file, one uses the -o [outfilepath] flag. If one were to use aic foo.pfil -o bar.exe, it would do the same as in the previous example, except the outfile would be named bar.exe.

If one wishes to go through the steps of compiler optimization on their file without compiling it, one would use the -po flag (preprocess only). This will output a file with the same name as would happen if it did not use the -po flag, except this file will be a .pfil as opposed to a .exe, and the .pfil will be optimized.

If one wishes to compile without proprocessing (which should only be done if the file was programmed in pure PFIL), one uses the -np flag (no preprocessing). This will compile the file without using the preprocessor. If one uses both -po and -np, an error results.

Some flags will override the compiling behavior of the program. -h brings up a syntax file, and -v displays the version of the compiler.

I must repeat that these are simply guidelines for command-line-based compilers, and should be taken with a grain of salt. Things not mentioned here may definitely be incorporated, though it is advised that all functionality described here be available in some form.
